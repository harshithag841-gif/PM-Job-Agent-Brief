Daily Trigger

Dual-trigger architecture with idempotent dedup guard. Double-firing is a safe no-op.

Primary — External Cron
cron-job.org → 08:15 IST
Free external scheduler hits GitHub’s POST /actions/workflows/daily.yml/dispatches REST API with a classic PAT (workflow scope). Independent of GitHub’s own scheduler reliability.
Fallback — GitHub Cron
53 2 * * * UTC → 08:23 IST
Native schedule: event on the workflow. Best-effort: GitHub has silently dropped runs multiple times since Jan 2026, so kept as a zero-cost safety net only.
Workflow
.github/workflows/daily.yml
20-min timeout, Ubuntu latest + Python 3.11 + Chrome stable, TZ=Asia/Kolkata. 10 secrets injected from GitHub’s encrypted vault. Commits jobs.db back to repo on success.
Idempotency guard. Both triggers land in the same workflow. A pre-flight _already_ran_today() check in main.py reads run_stats/history.csv — if today’s digest has already been sent, the run exits in under a second. Double-firing (both triggers land) is safe; the second run no-ops.
Why GitHub Actions, not a VPS? Zero idle cost, encrypted secrets vault, and I can read every run’s logs from any browser. A VPS would mean 24/7 compute for a workload that runs 20 minutes a day — and process supervision I’d have to maintain myself. Actions also gives me a free audit trail: every run is a linkable URL.

Step 2 — Deduplicate

Three-layer dedup: URL-based, semantic (company, role) keys, and a dual-pass cross-platform collapse across today’s scrape and yesterday’s queue.

2
SQLite jobs.db
deduplicator.py · jobs.db committed back to repo each run
seen_jobs
Every URL ever emailed, plus a dedup_key column that holds the normalized (company, role-bucket) pair. Both checked on every new scrape: URL match or key match → drop.
30-day rolling window
job_queue
Overflow beyond today’s top-50 cap, plus deferred-gate items for LLM re-scoring on lean days.
30-day expiry
score_cache
Persisted Groq LLM scores keyed by URL. Re-scoring skipped if a fresh score exists.
30-day window · saves LLM quota
Cross-platform collapse — dual-pass. One pass on today’s scrape pool (before scoring, reduces LLM cost) and a second pass on the merged pool (today’s qualifying + queue carryover). Both group by (normalized_company, role_bucket) and keep the higher-priority platform: LinkedIn → Hirist / IIMJobs / Instahyre → Indeed → Adzuna → Wellfound → WWR. Sub-qualifiers preserved so “Senior PM” and “Senior PM — Banking” stay distinct.
Why SQLite committed to the repo, not Postgres? The database is tiny (~400 KB), single-writer, and only needs to persist across daily runs. Committing it back to main after each run gives me free versioning, free backup, and zero infra — no managed DB, no connection strings, no ops. Git is the backup. If the file corrupts, I revert one commit.

Step 3 — Score (3-Layer Pipeline)

Deterministic filters first, LLM only for the ambiguous tail. LLM gate prevents overuse of Groq’s 30 RPM free tier.

3
Rules + Groq LLM
scorer.py · config.py (weights, thresholds, profile)
Layer 1 · Hard filters
Seniority keywords, freshness caps (LinkedIn 72h / others 7d), US-West-Coast geo drop. Runs inside each scraper before a job ever enters the pipeline.
scrapers/*.py + filters.py
Layer 2 · Graded rules
Auto-HIGH ladder (90/78/76/65/62) for strong matches. Auto-DROP for junior / non-PM / blocked geo. No API calls — covers ~70% of listings deterministically.
scorer.py → _layer2_score()
Layer 3 · Groq LLM
llama-3.3-70b-versatile, batched 10 jobs/call, 30s timeout. Smart gate skips LLM when strong pool already has 75 jobs; runs an opportunistic 50-job sample so nothing is blind. Deferred items drained from the queue on lean days.
30 RPM free tier · typical burn: 5 RPM
Why LLM only for the ambiguous tail, not for every job? Groq’s free tier caps at 30 RPM — enough only if I’m careful. Rules-only would miss the subtle mismatches LLMs catch in seconds; LLM-only would burn quota on jobs a keyword check could kill. Deterministic rules handle ~70% of decisions; the LLM earns its quota on the 30% where language ambiguity actually matters. Same logic as a senior reviewer: automate what’s obvious, think about what isn’t.

Runtime Modes

The orchestrator supports four modes. Scheduled production runs use defaults; the rest are developer / test affordances.

Production
(default)
Full pipeline. Sends email. Writes seen_jobs, job_queue, history.csv. Commits jobs.db back to main.
Dry run
--dry
Full scrape + score + rank. No email, no dedup writes, no commit-back. For verifying scraper output on demand.
Preview
--preview
Saves the digest HTML to /tmp/digest_preview.html instead of sending. No DB writes.
No-persist
NO_PERSIST=1
Sends the email so you see the real output, but skips every DB write (mark_sent, push_to_queue, commit-back, _already_ran_today guard). Intended for feature-branch end-to-end validation.

Operational Guardrails

Defensive measures added after an end-to-end audit. None block correct runs; all prevent known failure modes.

Excel formula-injection escape
Every scraped text field is passed through _safe_text() before write. Leading =, +, -, @ are neutralised so malicious job descriptions can’t execute as formulas when the workbook opens.
Bounded external calls
30-second timeout on every Groq LLM call. 30-second ThreadPoolExecutor wrap on every Resend email send. Prevents a stuck upstream from hanging the 20-minute workflow timeout.
Per-scraper isolation
Each scraper runs in its own try/except inside _run_scraper(). A crash in Wellfound doesn’t abort the run — the orchestrator logs a suspect state for that platform and continues. Empty-return vs. silent-block is distinguished by stdout scanning for error markers.
Anti-bot discipline
cloudscraper for Cloudflare-gated listings (WWR). ScraperAPI residential proxy for Wellfound. Apify managed actor for Indeed. Explicit failure modes kept in logs so silent 403s never look like empty results.
Source-level filtering
Every scraper enforces its own freshness cap (LinkedIn 72h, all others 7d), title regex, and (for WWR + Wellfound) a strict US-only policy that drops ambiguous remote roles unless a positive global-hiring signal is present. Stops junk from ever reaching the scoring layer.
Cross-platform dedup (dual-pass)
Same role at the same company on multiple platforms (LinkedIn vs Adzuna, today vs yesterday’s queue) collapses to one entry via a normalized (company, role-bucket) key. URL-based dedup misses these because each platform uses a different URL format. Pass 1 runs on today’s scrape pool, Pass 2 runs on the merged pool after queue load.
Data hygiene
Private repo. jobs.db carries URLs + scraped titles only, no PII. Credentials all in GitHub Secrets vault; .env + *.numbers git-ignored. Scoring reasons and user profile never logged in plaintext.

Fork This For Yourself

The architecture is portable. To adapt it for a different role, domain, or person, swap four things — the rest stays.

Swap 1 · The profile
Replace config.py’s user profile block with yours — resume keywords, years of experience, domain expertise, deal-breakers. The scorer reads this as ground truth.
Swap 2 · The platforms
Keep the scrapers that match your role’s market; drop the rest. Designers: add Dribbble, Coroflot. Engineers: add Hacker News Who’s Hiring, Arc. Add one scraper at a time — the _run_scraper() isolation makes failures safe.
Swap 3 · The scoring weights
Tune Layer 2’s ladder in scorer.py — which title patterns are auto-HIGH, which geos are auto-DROP, which seniority tier matters. A week of real output tells you where your thresholds are wrong.
Swap 4 · The delivery
Excel-in-email fits me. You may prefer a Notion page, a Slack DM, or a Telegram bot. The email_builder.py boundary is clean — one interface in, any channel out.
What doesn’t change: the dual-trigger, the 3-layer scoring philosophy, the SQLite-in-repo pattern, the dedup strategy, the free-tier discipline. Those are the load-bearing choices.
